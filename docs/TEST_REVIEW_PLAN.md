# 项目测试审计清单

> 目标：围绕“可信数据底座 + 可复现光储收益仿真引擎”，建立能暴露埋雷问题的测试体系。本文先列完整用例矩阵，再用代码逐步实现高价值用例。

## A. 配置与参数契约

- A001：`ESSParams` 默认值与 `data/params/ess_defaults.csv` 一致，容量、额定功率、效率、SOC、投资额单位正确。
- A002：`FinancialParams` 默认文件只要求当前有效字段：`r_discount`、`r_user_b1`、`r_user_b2`；B3 沿用 B2 分成口径。
- A003：`PVParams` 默认文件字段完整，容量、上网电价、单位造价、寿命、衰减率、曲线地区和类型可解析。
- A004：`WholesaleSettlementConfig` CSV 能往返解析为枚举对象，并能再展开为扁平字典。
- A005：保存函数不应在测试中污染全局参数文件；持久化测试必须使用临时文件或测试目录。
- A006：方案级 `ess_params` / `financial_params` / `private_overrides` 应在计算入口生效。

## B. 数据完整性与质量

- B001：现货电价文件按日期筛选后必须为 0~23 共 24 小时，单位转换后为元/kWh。
- B002：负荷文件按日期筛选后必须为 0~23 共 24 小时，`Load_real` 为非负且无缺失。
- B003：中长期合约与日前持仓文件按交易策略 `profile + selected_date` 筛选后必须为 0~23 共 24 小时。
- B004：行政分时电价 CSV 每个省份/月份/用电类别必须覆盖 0~23 小时，价格列至少有一列有效。
- B005：光伏曲线应支持 1440 分钟级或 24 小时级，聚合后为 24 点，且归一化出力非负。
- B006：具有地区属性的数据（电价、现货、统调负荷、光伏等）不应静默复用错误地区；无地区属性的典型用户负荷、交易策略持仓按 profile 复用。
- B007：数据文件需要登记可信度状态；当前不强制记录敏感来源，只记录 `mock/web_collected/official_api/manual_verified` 等内部可信级别。

## C. 电价与结算模型

- C001：M1 行政分时、M3 合同分时、M5 一口价均返回 24 点用户侧电价。
- C002：M4 现货联动应等于全月同小时日前价格均值。
- C003：M2 江苏模式当前若无专用配置，应有明确降级或明确失败，不应悄悄产生误导性结果。
- C004：批发购电广东型三部制分项与手工公式一致。
- C005：广西型、山西型附加项如保留，应先确认符合当前业务口径；阻塞成本、节点电价等不采用项已从当前计算口径清理。
- C006：负电价、极端高价、日前/实时价差反转场景应能稳定计算。

## D. 储能调度物理约束

- D001：任意商业模式下 `load_ESS` 长度为 24，且绝对值不超过额定功率。
- D002：SOC 始终位于 `[soc_min, soc_max]`。
- D003：充放电符号约定保持一致：充电为负，放电为正。
- D004：等效循环次数 = 总放电量 / 额定容量。
- D005：无光伏时 `load_grid = load_real - load_ESS`；有光伏时 `load_grid = load_real - load_ESS - pv_self_consumed`。
- D006：调度目标切换后，至少部分商业模式应产生不同调度或不同收益。
- D007：跨日连续 SOC、循环寿命、衰减开关目前未完全实现，应作为后续扩展测试。

## E. 光伏收益模型

- E001：PV 发电量 = 装机容量 × 归一化曲线。
- E002：PV 自发自用不超过 PV 发电量，也不超过储能调度后的正向净负荷。
- E003：PV 上网量 = PV 发电量 - 自发自用量，且非负。
- E004：PV 自用率位于 0~1。
- E005：PV 投资额、运维、IRR、NPV、回收期口径与参数单位一致。
- E006：容量为 0 时不返回 PV 投资结果；容量大于 0 时返回完整 PV 指标。

## F. API 与前端契约

- F001：`GET /api/options` 返回全部电价模式、商业模式和结算下拉项。
- F002：`GET /api/scenarios` 返回 `id/name` 列表，缺失方案返回 404。
- F003：`POST /api/calculate` 返回 24 点时序、概览、收益、投资、经济性评级。
- F004：API 响应中的 `pv_power`、`price_user`、`load_grid` 等前端依赖字段长度稳定。
- F005：计算缓存 key 应包含会影响结果的所有输入；目前未包含方案参数覆盖和全局参数版本，存在陈旧缓存风险。
- F006：`PUT /api/global-params` 会修改全局文件，测试必须隔离，否则容易污染工作区。

## G. 回归与工程卫生

- G001：全量测试不应修改仓库文件；若必须测试持久化，应使用临时目录。
- G002：测试断言应跟随 PRD 需求版本，避免旧字段导致噪音失败。
- G003：每个新增真实数据源至少有一个 smoke test，确认可读、可筛选、单位合理。
- G004：每个核心公式至少保留一个手工可展开的小样本测试。
- G005：后续容量优化、故障率、预测误差、蒙特卡洛等模块上线前，先补独立的基准场景测试。
