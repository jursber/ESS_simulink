# 2026-06-08 产品复盘与路线判断

> 目的：记录“第二版功能优化”后的工程 review、产品经理视角判断和下一阶段优先级。本文偏产品与架构口径，方便多电脑开发时延续上下文。

## 当前产品定位

本项目不应被理解为一个简单的储能收益计算器，而应被理解为一个面向中国电力市场的源网荷储售一体化策略仿真与决策系统。

理想形态是：

- 底层：可信、可追溯、可替换的数据资产。
- 中层：可解释、可测试、可版本化的市场、调度、财务、设备模型。
- 上层：单方案分析、多方案对比、深度精算、方案管理、策略解释和策略推荐。

系统最终要回答的不只是“收益是多少”，而是“在当前边界条件下，为什么这个策略更优，它对哪些数据、规则、设备状态最敏感，以及它在真实运行中可能偏离多少”。

## 本轮完成后的工程 review

### 已经明显改善的部分

- 多方案对比、方案管理、模型管理、文档系统和数据资产总览从占位推进到可用状态。
- `ScenarioConfig.ess_params`、`financial_params` 和 `private_overrides` 已进入运行时计算，不再只是存储字段。
- `/api/calculate` 缓存 key 已加入方案内容指纹和全局参数文件版本，降低参数变化后返回旧结果的风险。
- 模型超参草稿支持保存和回显，并对模型 ID 写入路径做了安全限制。
- 旧的非 `.v4` 前端文件已删除，当前入口更清晰。
- 测试结果提升到 `172 passed, 2 xfailed`，剩余 xfail 为江苏模式配置迁移/降级相关，不影响当前河南主链路。

### 仍然埋雷的问题

#### P0：方案保存语义不完整

当前顶部“保存”还不是完整方案快照。它主要保存基础选择项，且前端仍存在商业模式写死为 `B1` 的情况。

风险：用户以为保存的是当前完整仿真状态，但再次加载或参与多方案对比时，实际计算边界可能已经不同。

需要尽快定义并实现：

```text
方案 = 基础元信息
     + 仿真模式
     + 数据集引用
     + 全局参数版本
     + 前端当前选择
     + 私有参数覆盖
     + 模型版本/超参
     + 结果摘要
```

#### P0：单方案槽位到多方案对比仍可能丢失边界条件

单方案槽位中记录了批发侧规则、合约曲线 profile、日前曲线 profile 等信息，但多方案对比当前主要使用方案、电价模式、商业模式。

风险：单方案页看到的计算结果，与多方案对比页同名方案的结果可能不一致。

#### P1：模型超参还没有真正驱动算法

模型管理页现在可以保存和回显草稿，但调度引擎没有读取这些超参。

风险：页面会让用户觉得模型已经可调，而实际仍是固定算法参数。当前前端已显示“草稿已保存”，但后续必须接入算法运行时。

#### P1：缓存 key 还没有覆盖全部数据资产版本

当前缓存 key 覆盖方案内容和 `data/params/*.csv`。但现货价格、负荷、光伏曲线、交易策略等底层数据更新后，缓存仍可能不感知。

后续真实数据大量接入后，需要升级为数据集版本签名。

#### P1：典型日与全年数据的产品语义仍需落地

已经讨论出“快速预览 / 深度精算”双模式，但目前后端计算仍主要是典型日/24 点逻辑。全年 8760 或 8760*4 数据还没有进入正式仿真模式。

## 产品经理视角的路线判断

### 第一优先级：方案快照契约

下一步不要优先继续堆新页面，而是先把“方案到底是什么”定死。

建议新增一个正式的 `ScenarioSnapshot` 概念，用于支撑：

- 保存方案
- 加载方案
- 多方案对比
- 方案复制
- 结果复现
- 深度精算任务
- 未来策略优化

只要方案契约不稳，多方案对比、方案管理、深度精算都会不断返工。

### 第二优先级：仿真模式

正式实现两个模式：

- 快速预览：典型日/24 点，参数变化后快速反馈。
- 深度精算：全年 8760 或 8760*4，计算更慢，但用于正式结论。

保存方案时应保存仿真模式；多方案对比也应明确每个结果来自哪种模式。

### 第三优先级：数据资产版本化

随着真实光伏出力、真实现货价格持续接入，数据资产会成为项目核心壁垒。

需要引入：

- 数据集 ID
- 数据版本号
- 数据可信度
- 是否 mock
- 时间范围
- 适用 region/profile
- 行数、时间粒度、缺失率等质量指标

后续每一次仿真结果都应能追溯到具体数据版本。

### 第四优先级：结果解释系统

一个高级仿真系统不能只输出表格。它应该解释：

- 哪个方案更优。
- 优势来自储能套利、用户节费、售电利润还是光伏消纳。
- 结果对现货价差、负荷峰谷、PV 曲线、储能效率、合同持仓等哪个因素最敏感。
- 哪些结果可能是 mock 数据导致的，不适合用于正式结论。

这会让系统从“计算器”变成“决策助手”。

### 第五优先级：真实运行因素

用户已经明确希望未来更真实。建议逐步加入：

- 储能故障率/可用率。
- 检修计划。
- PCS/电池限功率。
- 容量衰减曲线。
- 实际执行偏差。
- 计量/结算滞后。
- 策略执行失败期间收益扣减。

这些因素不必马上做，但应在数据结构中预留位置。

### 第六优先级：策略优化与容量配置

“储能/光伏应该配置多大容量才能收益最好”现在不是当前需求，但它一定会成为高级功能。

建议等以下基础稳住后再做：

- 方案快照契约稳定。
- 全年深度精算可用。
- 数据版本可追溯。
- 模型版本和超参可追溯。
- 结果解释和敏感性分析具备雏形。

否则容量优化可能会建立在不稳定边界上，得到看似精确但不可复现的结果。

## 建议的近期 TODO

1. 实现 `ScenarioSnapshot`，替代当前零散保存逻辑。
2. 修复顶部保存动作，保存完整当前选择和关键私有参数。
3. 让单方案槽位传入多方案对比时保留批发侧规则、合约 profile、日前 profile。
4. 把模型超参草稿接入 `run_dispatch()` 或算法注册表。
5. 把缓存签名扩展到现货、负荷、PV、交易策略等数据资产版本。
6. 在多方案对比结果中显示“快速预览/深度精算”和数据可信度。
7. 为深度精算设计任务状态：待运行、运行中、完成、失败、过期。

## 一句话判断

这个项目的雄心不应是“算一笔储能收益账”，而是建立一个足够真实、可追溯、可解释、可扩展的电力市场数字实验室。下一阶段最重要的不是更多页面，而是把方案、数据、模型和结果四件事的契约彻底打牢。
