# 仿真时域与性能设计

## 背景

当前系统的主要计算方式是：每个省份或场景选择 1 天典型数据，用户选择计算 1 个月或 1 年时，按 `典型日结果 * 天数` 外推。

这个方式的优点是交互体验很快，参数改动后可以立即得到反馈；缺点是会浪费已有的 8760 小时或 35040 点十五分钟级数据资产，也无法真实反映季节性、工作日/周末差异、光伏年内波动、现货价格尖峰、需量电费触发时刻、SOC 跨日连续性等问题。

用户明确担心：如果直接改成复杂长序列算法，每改一个小参数都跑一次 8760/35040 仿真，体验可能很糟糕。

## 核心判断

系统不应在“典型日秒级反馈”和“8760 全量真实仿真”之间二选一，而应把仿真能力拆成多层：

1. 快速预览层：用于调参时高频反馈，目标是秒级返回。
2. 精确复算层：用于用户停止调参或点击复算后，使用真实月份/年度序列，目标是数秒到十几秒内返回。
3. 批量深算层：用于多场景、多省份、多策略组合评估，可异步执行，目标是结果可信而不是即时。

## 推荐模式

### 1. typical_day 快速预览

保留当前典型日模式，不废弃。

适用场景：

- 用户拖动储能功率、容量、效率、电价参数时的即时反馈。
- 前端页面上的指标预览。
- 早期粗筛策略或参数范围。

限制：

- 不能作为严肃年度收益结论。
- 不能准确表达季节性、尖峰电价、需量电费、跨日 SOC、真实光伏曲线等。

### 2. representative_days 代表日/代表周

从真实 8760/35040 数据中抽取或聚类出少量代表日/代表周，并给每个样本分配权重。

适用场景：

- 用户希望比典型日更可信，但仍需要较快交互。
- 年度测算的中间档模式。

建议：

- 不应只给用户暴露大量日期让用户自己选。
- 应由系统根据真实数据生成若干代表日，例如高价日、低价日、高光伏日、低光伏日、典型工作日、典型周末、极端尖峰日等。

### 3. date_range / monthly / annual 真实序列复算

严肃测算应支持按日期范围读取真实序列，并对全周期逐点仿真。

适用场景：

- 月度收益结算口径。
- 年度投资收益评估。
- 比较不同策略真实表现。
- 需量/容量电价相关测算。
- 使用真实光伏出力、真实日前/实时现货价格。

实现建议：

- 初期可先按“逐日滚动 + 汇总”实现，降低改造风险。
- 后续再支持连续 SOC 跨日传递。
- 15 分钟级数据优先服务于需量电费、光伏波动和现货尖峰捕捉。

## 性能原则

1. 参数编辑不直接触发年度全量仿真。
2. 小参数变化先触发快速预览。
3. 用户停止输入、点击复算、保存场景或进入结果页时，再触发精确复算。
4. 精确结果应带有状态，例如 `preview`、`recalculating`、`final`。
5. 年度/多场景批量计算应支持异步任务和缓存。

## 缓存原则

仿真缓存 key 必须包含所有会影响结果的输入，包括但不限于：

- 场景参数。
- 储能参数。
- 财务参数。
- 用户负荷 profile。
- 光伏数据集及版本。
- 现货电价数据集及版本。
- 统调负荷数据集及版本。
- 中长期/日前持仓 profile。
- 仿真时域、分辨率、起止日期。
- 算法版本。

当前项目已修复 API 缓存 key 的主要风险：`/api/calculate` 会纳入方案内容指纹和全局参数文件版本。后续引入全年深度精算时，还需要继续把仿真模式、数据集版本和算法版本纳入缓存签名。

## 前端交互建议

前端不应让用户在大量真实日期中手动选择到崩溃，而应把复杂度收进数据层和算法层。

建议暴露为：

- 快速预览：典型日/代表日，适合调参。
- 月度复算：选择月份。
- 年度复算：选择年份。
- 高级模式：选择自定义日期范围、分辨率、数据集版本。

页面上的结果需要区分“预览结果”和“精确结果”，避免用户误把典型日外推当作最终结论。

## 保存语义

顶部“保存”按钮建议保存的是“测算方案”，而不是单纯保存某一次计算结果。

测算方案应包含：

- 用户选择的业务场景。
- 仿真模式，例如快速预览或深度精算。
- 起止日期、月份、年份或典型日选择。
- 数据分辨率，例如小时级或十五分钟级。
- 用户负荷 profile。
- 光伏数据集及版本。
- 现货电价数据集及版本。
- 统调负荷数据集及版本。
- 中长期/日前持仓 profile。
- 储能参数。
- 电价、需量/容量电费、财务参数等输入。
- 策略参数。
- 数据可信度标记。
- 算法版本。

保存时可以同时关联“最近一次计算结果快照”，但该快照必须带状态：

- `preview_result`：快速预览结果，只用于调参参考。
- `final_result`：深度精算完成后的正式结果。
- `stale`：方案参数已改变，之前的结果已经过期。

推荐交互：

- 快速预览 Tab 中，参数变化后自动计算并更新 `preview_result`。
- 深度精算 Tab 中，参数变化后只把正式结果标记为 `stale`，不自动跑年度全量仿真。
- 用户点击“重新计算”后，生成新的 `final_result`。
- 用户点击顶部“保存”时，保存当前方案输入，并保留当前可用的结果快照及其状态。

这样做的好处是：用户可以随时保存工作进度；系统也不会把过期结果误认为正式结论。

## 后续改造路线

第一阶段：

- 保留当前 24 点典型日能力。
- 给场景模型增加仿真模式、起止日期、分辨率字段。
- 数据 loader 支持按 region + date_range 读取多日数据。
- 新增测试保护典型日逻辑不退化。

第二阶段：

- 增加按日期范围的逐日滚动仿真 wrapper。
- 月度/年度结果先由每日结果汇总得到。
- 明确结果状态和缓存 key。

第三阶段：

- 支持连续长序列 SOC。
- 支持代表日/代表周生成。
- 支持批量深算任务队列。

第四阶段：

- 在前端引入“快速预览/精确复算/批量深算”的交互区分。
