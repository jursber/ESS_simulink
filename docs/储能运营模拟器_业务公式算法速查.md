# 储能运营模拟器：业务、公式与算法速查

> 最后更新：2026-06-10  
> 用途：帮助快速核对系统当前做了什么、为什么这样做、哪些口径被暂时忽略，以及核心公式是否符合业务理解。  
> 当前版本重点：单日 24 小时简单模拟；储能调度已切换为单储能峰谷套利 MILP 基准模型。

---

## 一、业务说明：系统如何连接、有哪些模式、各模式关注什么

### 1.1 系统主链路

当前计算入口是 `POST /api/calculate`，后端主流程如下：

```text
方案配置 ScenarioConfig
  -> calculator.calculate()
  -> 读取曲线与参数
     - 用户负荷
     - 日前/实时现货价格
     - 中长期合约持仓
     - 日前申报/出清电量
     - 行政/合同分时电价
     - 储能、光伏、财务参数
     - 批发结算参数
  -> compute_user_price() 生成用户侧 24 小时电价 P_user
  -> run_dispatch() 生成储能调度、光伏消纳、电网结算电量
  -> compute_wholesale_purchase_cost() 计算售电公司批发购电成本
  -> 汇总用户、售电公司、储能、光伏和组合收益
  -> CalculateResponse 返回前端
```

当前系统的核心假设是：先用单日 24 小时曲线做快速预览，月度和年度指标暂按典型日外推。它适合调参和业务逻辑验证，不等同于严格年度收益结论。

### 1.2 参与主体

| 主体 | 当前系统中关注的核心问题 | 主要收入/成本口径 |
|---|---|---|
| 用户 | 用电账单是否下降，分享后净得多少 | 无储能账单、含储能账单、节省金额、用户分成 |
| 售电公司 | 零售收入与批发购电成本之间的利润 | 用户侧售电收入 - 批发市场购电成本 |
| 储能运营商/投资方 | 储能套利收益、年净现金流、IRR、NPV、回收期 | 用户节省中的储能分成，或组合模式下的组合利润 |
| 光伏系统 | 自发自用、余电上网、光伏投资收益 | 自用收益、上网收益、光伏 IRR/NPV/回收期 |
| 电网/批发市场 | 中长期、日前、实时和分摊项如何结算 | 中长期电费、日前偏差、实时偏差、月度常数项 |

### 1.3 电价模式

| 模式 | 代码 | 当前含义 | 主要关注点 |
|---|---|---|---|
| 行政分时电价 | M1 | 按行政 TOU 表生成用户侧电价 | 常规峰谷套利、用户侧节费 |
| 江苏系数模式 | M2 | 以基准价乘分时系数；无配置时回退到 M1 | 旧口径兼容，当前不是重点 |
| 合同分时电价 | M3 | 按合同 TOU 表生成用户侧电价 | 用户合同价下的储能套利 |
| 现货联动 | M4 | 指定现货曲线时，按联动比例 + 固定价生成用户侧价格；否则按月度日前均值 | 现货价格向零售价格传导 |
| 一口价 | M5 | 24 小时统一用户侧电价 | 验证无峰谷价差时的调度稳定性 |
| 合同+现货联动 | M4-contract | UI 扩展选项，运行时映射到 M4 | 合同曲线与现货联动的混合零售口径 |

M4 当前有一个重要口径：如果选择了典型现货曲线，用户侧价格不是简单等于日前价格，而是：

```text
P_user[t] = spot_link_ratio * P_base[t] + (1 - spot_link_ratio) * spot_fixed_price
```

其中 `P_base` 可选日前或实时价格。这个口径会影响储能调度目标。

### 1.4 商业模式

| 模式 | 代码 | 当前业务解释 | 调度关注点 |
|---|---|---|---|
| 用户自投储能 | B1 | 用户拥有或等同拥有储能，储能用于降低自身用电账单 | 用户侧峰谷套利 |
| 售电公司视角 | B2a | 售电公司关注零售收入与批发购电成本 | 实时/零售价差、批发结算成本 |
| 储能运营商视角 | B2b | 第三方储能运营商关注自身收益 | 用户侧节费分成 |
| 用户视角 | B2c | 第三方模式下用户关注净节省 | 用户分成后净收益 |
| 储售一体 | B3a | 售电与储能合并看组合利润 | 组合收益最大化 |
| 储售一体下用户 | B3b | 用户侧仍关注分成后收益 | 用户侧电价 |
| 社会福利 | B4 | 从系统整体价值观察 | 实时价格信号 |

当前 `run_dispatch()` 仍统一用有效价格 `P_eff` 表达不同商业模式下的调度目标，但本期新增的单储能 MILP 模型重点服务“单储能峰谷套利”基准口径。

### 1.5 系统组合模式

当前系统支持三类资产组合：

| 组合 | 含义 | 当前处理方式 |
|---|---|---|
| 仅负荷 | 没有储能、没有光伏 | 只计算用户账单和售电公司结算 |
| 负荷 + 储能 | 储能从电网充电，放电抵扣用户负荷 | 本期重点，禁止无光伏场景反送电 |
| 负荷 + 光伏 + 储能 | 光伏先自发自用，储能可参与调度，允许净上网 | 净上网按光伏上网电价结算；暂不追溯电量来源 |

### 1.6 本期新增口径

本期明确把“储能反送电”纳入调度约束：

1. 无光伏时，储能向电网反送电无收益，且算法必须禁止反送电。
2. 无光伏时必须满足：

```text
load_ESS[t] <= load_real[t]
```

其中 `load_ESS[t] > 0` 表示储能放电。

3. 有光伏时，允许净上网，当前按：

```text
上网收益 = 反送电量 * 光伏上网电价
```

4. 本期暂不判断夜间反送是否来自光伏，也暂不实现“储能反送明显超过光伏功率”的罚款规则。

## 二、计算公式：电价、结算、收益与投资指标

### 2.1 关键符号

| 符号 | 含义 | 单位 |
|---|---|---|
| `t` | 小时，0 到 23 | h |
| `L[t]` | 用户真实负荷 `load_real` | kWh |
| `E_ess[t]` | 储能净出力 `load_ESS`，放电为正、充电为负 | kWh |
| `SOC[t]` | 小时末 SOC | 0-1 |
| `P_user[t]` | 用户侧电价 | 元/kWh |
| `P_da[t]` | 日前价格 | 元/kWh |
| `P_rt[t]` | 实时价格 | 元/kWh |
| `P_eff[t]` | 调度有效价格信号 | 元/kWh |
| `PV[t]` | 光伏发电量 | kWh |
| `PV_self[t]` | 光伏自发自用电量 | kWh |
| `G[t]` | 与电网结算的净购电量 `load_grid` | kWh |
| `G_import[t]` | 从电网购电量 | kWh |
| `G_export[t]` | 向电网反送电量 | kWh |
| `r_user` | 用户分成比例 | - |

### 2.2 用户侧电价

#### M1：行政分时电价

```text
P_user[t] = tariff_admin(period(t))
```

由行政分时电价表按小时查表获得。

#### M2：江苏系数模式

如果有江苏模式配置：

```text
P_user[t] = p_base * coefficient(period(t))
```

如果没有配置，当前回退到 M1。

#### M3：合同分时电价

```text
P_user[t] = tariff_contract(period(t))
```

由合同分时电价表按小时查表获得。

#### M4：现货联动

旧通用口径：

```text
P_user[t] = monthly_average(P_da[:, t])
```

当前典型现货曲线口径：

```text
P_user[t] = alpha * P_base[t] + (1 - alpha) * P_fixed
```

其中：

```text
alpha = spot_link_ratio
P_fixed = spot_fixed_price
P_base = P_da 或 P_rt
```

#### M5：一口价

```text
P_user[t] = P_flat
```

当前默认 `P_flat = 0.55` 元/kWh。

### 2.3 商业模式有效价格信号

当前调度层把不同商业模式映射为 `P_eff[t]`：

| 商业模式 | 有效价格信号 |
|---|---|
| B1 用户自投储能 | `P_eff[t] = P_user[t]` |
| B2a 售电公司 | `P_eff[t] = P_rt[t] - P_user[t]` |
| B2b 储能运营商 | `P_eff[t] = P_user[t]` |
| B2c 用户 | `P_eff[t] = P_user[t]` |
| B3a 储售一体 | `P_eff[t] = P_rt[t] - r_user * P_user[t]` |
| B3b 储售一体下用户 | `P_eff[t] = P_user[t]` |
| B4 社会福利 | `P_eff[t] = P_rt[t]` |

直观理解：

```text
P_eff[t] 越低，越适合充电
P_eff[t] 越高，越适合放电
```

### 2.4 光伏计算

```text
PV[t] = PV_cap * PV_curve[t]
```

当前光伏自用先覆盖用户原始负荷：

```text
PV_self[t] = min(PV[t], max(0, L[t]))
```

本期调度模型中，储能本地可消纳放电上限按光伏自用后负荷计算：

```text
local_absorption_limit[t] = max(0, L[t] - PV[t])
```

说明：这是一种保守口径，用于避免储能挤占光伏自用。后续如需更精细，应区分“光伏直供负荷、光伏充储、储能放电、储能反送”的电量来源。

### 2.5 电网结算电量

```text
G[t] = L[t] - E_ess[t] - PV_self[t]
```

拆分为：

```text
G_import[t] = max(0, G[t])
G_export[t] = max(0, -G[t])
```

无光伏时，算法约束保证：

```text
G_export[t] = 0
```

有光伏时，当前允许：

```text
G_export[t] > 0
```

并按光伏上网电价计算收益。

### 2.6 用户账单与节省

无储能账单：

```text
Bill_no_ess = Σ L[t] * P_user[t]
```

含储能/光伏账单：

```text
Export_revenue = Σ G_export[t] * PV_feed_in_tariff
Bill_with_ess = Σ G_import[t] * P_user[t] - Export_revenue
```

用户侧节省：

```text
User_savings = Bill_no_ess - Bill_with_ess
```

用户净得：

```text
User_net = User_savings * r_user
```

储能侧收益：

```text
ESS_revenue = User_savings * (1 - r_user)
```

注意：当前 `r_user` 是收益分成参数，不是电价折扣。

### 2.7 售电公司批发购电成本

当前售电公司批发购电成本在 `compute_wholesale_purchase_cost()` 中计算，核心是中长期、日前、实时三段式。

#### 广东型三部制

```text
C_mlt = Σ Q_LT[t] * P_LT[t]
C_da  = Σ (Q_DA[t] - Q_LT[t]) * P_DA[t]
C_rt  = Σ (Q_act[t] - Q_DA[t]) * P_RT[t]

Purchase_cost = C_mlt + C_da + C_rt + C_monthly_constant
```

其中：

```text
Q_act[t] = G[t]
```

当前代码把与电网结算的净负荷作为实际购电量输入。

#### 广西型

```text
C_mlt = Σ Q_LT[t] * (P_LT[t] + P_DA[t] - P_ref[t])
C_da  = Σ (Q_DA[t] - Q_LT[t]) * P_DA[t]
C_rt  = Σ (Q_act[t] - Q_DA[t]) * P_RT[t]

Purchase_cost = C_mlt + C_da + C_rt + C_month_smooth + C_monthly_constant
```

#### 山西型

当前先采用广东型三部制框架，并额外加入可配置附加项：

```text
Purchase_cost = C_mlt + C_da + C_rt + C_shanxi_addon + C_monthly_constant
```

完整山西差价合约细则尚未展开。

#### 山东

当前暂按广东型三部制处理，属于待补充口径。

### 2.8 售电公司收入与利润

售电公司零售收入：

```text
Retail_revenue = Bill_with_ess
```

售电公司利润：

```text
Retail_profit = Retail_revenue - Purchase_cost
```

说明：这里的 `Bill_with_ess` 已经扣除了有光伏场景下的上网收益，后续如区分用户、光伏业主、售电公司之间的上网收益归属，需要进一步拆账。

### 2.9 组合收益与社会福利

组合利润：

```text
Combined_profit = Retail_profit + ESS_revenue
```

如果没有售电公司主体，则主要体现为储能侧收益。

社会福利：

```text
Total_welfare = User_net + Retail_profit + ESS_revenue
```

### 2.10 投资指标

储能初始投资：

```text
ESS_investment = ESS_cap_kWh * unit_cost_yuan_per_Wh * 1000
```

年运维成本：

```text
ESS_OM_annual = ESS_investment * r_om
```

年现金流：

```text
Annual_cashflow = selected_daily_profit * 365 - ESS_OM_annual
```

不同商业模式下 `selected_daily_profit` 不同：

| 模式 | 年现金流取值 |
|---|---|
| B1 | 储能收益 |
| B2a/B2b/B2c | 储能收益 |
| B3a/B3b | 组合利润 |
| B4 | 组合利润 |

静态回收期：

```text
Payback_years = ESS_investment / Annual_cashflow
```

NPV：

```text
NPV = -ESS_investment + Σ Annual_cashflow * capacity_ratio[y] / (1 + r_discount)^y
```

其中：

```text
capacity_ratio[y] = ((1 - r_degrade) ^ y)
```

IRR：用二分法求使 NPV 等于 0 的折现率。

等效循环：

```text
Equivalent_cycles = Σ max(0, E_ess[t]) / ESS_cap_kWh
```

### 2.11 光伏投资指标

光伏初始投资：

```text
PV_investment = PV_cap_kWp * PV_unit_cost_yuan_per_Wp * 1000
```

光伏日收益：

```text
PV_self_daily = Σ PV_self[t] * P_user[t]
PV_feed_in_daily = Σ PV_fed_in[t] * PV_feed_in_tariff
PV_daily_revenue = PV_self_daily + PV_feed_in_daily
```

光伏年现金流：

```text
PV_annual_cashflow = PV_daily_revenue * 365 - PV_investment * PV_r_om
```

光伏容量保持率：

```text
Year 1: 1 - r_degrade_first
Year y: (1 - r_degrade_first) * (1 - r_degrade)^(y - 1)
```

光伏 NPV/IRR 与储能类似。

## 三、算法：名称、目标、方法、逻辑、优劣与忽略项

### 3.1 当前调度算法清单

| 算法 ID | 名称 | 当前状态 | 用途 |
|---|---|---|---|
| `greedy_window` | 贪心滑窗搜索 | 保留，旧算法 | 快速启发式调度 |
| `peak_valley_milp` | 单储能峰谷套利 MILP | 本期新增并接入 `run_dispatch()` | 单储能峰谷套利基准模型 |

### 3.2 旧算法：`greedy_window`

#### 调度目标

```text
max Σ E_ess[t] * P_eff[t]
```

#### 算法逻辑

1. 按价格从低到高选充电候选池。
2. 按价格从高到低选放电候选池。
3. 在候选池内枚举不同充电小时数、放电小时数和滑动窗口组合。
4. 对每组候选小时按 0 到 23 小时顺序模拟 SOC。
5. 选收益最高的组合。

#### 优点

- 逻辑直观，速度快。
- 对 24 小时典型日足够轻量。
- 便于解释“低价充、高价放”。

#### 缺点

- 不是严格全局最优。
- 候选池和窗口参数会影响结果。
- 不天然处理“无光伏禁止反送电”这种电网结算边界。
- 平价或复杂价格形态下可能给出不够稳健的结果。

### 3.3 新算法：`peak_valley_milp`

#### 使用场景

本期用于单储能简单峰谷套利，尤其适合：

- 用户侧分时价套利；
- 单日 24 小时典型曲线；
- 需要强制物理约束和反送电边界；
- 需要在任意价格数据下稳定运行。

#### 决策变量

对每个小时 `t`：

```text
charge[t]             >= 0  储能充电量
discharge_to_load[t]  >= 0  放电供用户负荷
discharge_export[t]   >= 0  放电/净上网电量
soc[t]                      SOC
is_charging[t] ∈ {0,1}      充电状态变量
```

输出到系统的储能净出力：

```text
E_ess[t] = discharge_to_load[t] + discharge_export[t] - charge[t]
```

#### 调度目标

模型最小化成本，等价于最大化套利收益：

```text
max Σ discharge_to_load[t] * P_eff[t]
  + Σ discharge_export[t] * P_export[t]
  - Σ charge[t] * P_eff[t]
```

实现中使用最小化形式：

```text
min Σ charge[t] * P_eff[t]
  - Σ discharge_to_load[t] * P_eff[t]
  - Σ discharge_export[t] * P_export[t]
  + ε * throughput
```

其中 `ε = 1e-6`，只用于同收益场景下减少无意义充放电，不改变有明显价差时的主决策。

#### 物理约束

功率约束：

```text
0 <= charge[t] <= P_max
0 <= discharge_to_load[t] <= P_max
0 <= discharge_export[t] <= P_max
```

SOC 约束：

```text
SOC_min <= SOC[t] <= SOC_max
```

SOC 递推：

```text
SOC[t+1] = SOC[t]
         + charge[t] * eta / ESS_cap
         - discharge_to_load[t] / (eta * ESS_cap)
         - discharge_export[t] / (eta * ESS_cap)
```

初始 SOC：

```text
SOC[0] = soc_initial
```

当前没有强制日末 SOC 回到日初 SOC。

#### 充放互斥约束

```text
charge[t] <= P_max * is_charging[t]
discharge_to_load[t] + discharge_export[t] <= P_max * (1 - is_charging[t])
```

这个约束避免同一小时既充又放。

#### 无光伏反送电约束

无光伏时：

```text
discharge_export[t] = 0
discharge_to_load[t] <= load_real[t]
```

因此：

```text
G[t] >= 0
```

即储能不会向电网反送电。

#### 有光伏上网收益约束

有光伏时：

```text
discharge_export[t] >= 0
P_export[t] = PV_feed_in_tariff
```

允许：

```text
G[t] < 0
```

并按光伏上网电价计入收益。

### 3.4 新算法优点

1. **全局最优性更强**：峰谷套利是线性目标 + 线性约束，用 MILP 能稳定求解当前 24 小时问题。
2. **边界清晰**：无光伏禁止反送电、有光伏允许上网，直接写入约束。
3. **数据鲁棒**：零容量、零功率、NaN/异常价格、平价、小价差、负电价等场景都能返回结果。
4. **可解释**：每个变量都能映射到充电、放电、自用、上网、SOC。
5. **可扩展**：后续可以继续加日末 SOC、最大循环次数、需量电费、罚款、光伏来源追溯等约束。

### 3.5 新算法缺点

1. **比贪心更重**：虽然 24 小时规模很小，但 MILP 仍比简单贪心复杂。
2. **当前仍是单日模型**：没有处理跨日 SOC 连续性，典型日外推会高估或低估真实年度收益。
3. **当前未做电量来源追溯**：有光伏时，净上网按光伏上网电价结算，但没有区分这部分电量到底来自光伏还是电网充入储能。
4. **暂未纳入需量电费**：对工商业用户，需量/容量电费可能显著影响储能调度。
5. **暂未纳入循环寿命约束**：当前只计算等效循环，没有把循环寿命作为硬约束或成本项。
6. **售电公司复杂结算还未成为调度内生目标**：批发购电成本在收益侧计算，后续如果要让调度直接服务售电公司组合收益，需要把中长期/日前/实时偏差更深地并入优化目标。

### 3.6 本期明确忽略项

以下不是遗漏，而是本期故意不实现：

| 忽略项 | 原因 | 后续建议 |
|---|---|---|
| 夜间储能反送是否算光伏上网 | 需要电量来源追踪，超出本期简单模型 | 增加“电池电量来源”状态变量 |
| 储能反送超过光伏功率的罚款 | 需要政策/合同罚则参数 | 作为惩罚项加入目标函数 |
| 日末 SOC 回到日初 | 单日预览阶段先观察套利潜力 | 增加 `terminal_soc_policy` |
| 多日/月度/年度连续仿真 | 当前主链路是 24 小时典型日 | 新增 date_range wrapper 和跨日 SOC |
| 需量电费 | 需要最大需量窗口和计费规则 | 加入峰值需求变量 |
| 循环寿命硬约束 | 当前只算等效循环 | 加入日循环上限或吞吐成本 |
| 光伏充储与储能放电来源区分 | 当前只有净电量，没有能量来源账本 | 建立 PV->load、PV->battery、grid->battery 分流变量 |

### 3.7 当前测试方案

本期新增测试围绕调度目标，而不是只测代码能运行：

| 测试目标 | 验证内容 |
|---|---|
| 低价充高价放 | 低价时段应充电，高价时段应放电，收益为正 |
| 无光伏禁止反送 | 任意时刻 `储能放电 - 用户负荷 <= 0` |
| 平价不循环 | 价格完全相同时不应出现无意义充放电 |
| 小价差不循环 | 价差不足以覆盖效率损耗时不应伪套利 |
| 异常参数空调度 | 零容量、零功率等参数返回全 0 调度 |
| 有光伏允许上网 | 有上网电价时允许净上网并产生收益 |
| run_dispatch 无光伏并网约束 | 最终 `load_grid >= 0` |
| run_dispatch 有光伏上网收益 | `load_grid < 0` 时按上网电价改善账单 |

已验证命令：

```bash
pytest tests/test_peak_valley_dispatch.py
pytest tests/test_peak_valley_dispatch.py tests/test_calculator.py -q -k "not spot_curve_id"
```

说明：`spot_curve_id` 相关旧测试当前反映的是 M4 现货联动价格口径与旧断言不一致，不属于本期单储能调度模型引入的问题。

## 四、快速校对清单

如果后续检查模型是否理解错业务，可以优先看这几条：

1. 储能充电一定增加电网购电量。
2. 无光伏时储能不能反送电，放电最多抵扣当时用户可消纳负荷。
3. 有光伏时当前允许净上网，暂按光伏上网电价结算。
4. 当前单日模型没有强制日末 SOC 回到日初。
5. 当前没有追溯上网电量来源，因此夜间储能反送暂不会被单独识别。
6. 当前收益分成使用 `r_user`，用户拿 `User_savings * r_user`，储能拿 `User_savings * (1-r_user)`。
7. 当前批发结算已经能算广东/广西/山西/山东抽象口径，但山西、山东仍是简化模型。
8. 当前结果适合“单日典型曲线快速预览”，不应直接当作严谨年度收益结论。
